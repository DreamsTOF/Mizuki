## 1. SPI 模式（Service Provider Interface）

### 什么是 SPI 模式？

**定义：** SPI（Service Provider Interface）是一种**服务发现机制**，允许框架/库定义接口规范，由第三方提供具体实现，并在运行时动态发现和加载这些实现。

**核心特征：**
1. **接口与实现分离**：框架只定义接口，不关心具体实现
2. **动态发现**：运行时自动发现并加载可用的实现
3. **可插拔**：可以随时添加、替换实现，无需修改框架代码
4. **解耦**：框架不依赖具体实现，实现者也不需要修改框架

**为什么这是 SPI 模式而不是策略模式？**
- 策略模式：调用方**主动选择**使用哪个策略，策略通常在编译时就确定了
- SPI 模式：**被动发现**可用的实现，实现可以在运行时动态添加，调用方不需要知道具体实现类

### 实现位置
`cn.dreamtof.core.context.ThreadContextCopier`

### 示例代码

```java
public interface ThreadContextCopier {
    Object capture();           // 捕获父线程上下文
    void restore(Object contextSnapshot);  // 恢复到子线程
    void clear();               // 清理子线程上下文
}
```

### SPI 注册与调用
`cn.dreamtof.core.context.ContextPropagator`

```java
private static final List<ThreadContextCopier> COPIERS = new CopyOnWriteArrayList<>();

public static void registerCopier(ThreadContextCopier copier) {
    COPIERS.add(copier);
}

@Override
public Runnable decorate(Runnable runnable) {
    List<Object> snapshots = new ArrayList<>(COPIERS.size());
    for (ThreadContextCopier copier : COPIERS) {
        snapshots.add(copier.capture());
    }
    return () -> {
        for (int i = 0; i < COPIERS.size(); i++) {
            COPIERS.get(i).restore(snapshots.get(i));
        }
        // ...
    };
}
```

## 2. 注册表模式（Registry Pattern）

### 什么是注册表模式？

**定义：** 注册表模式是一种**全局对象管理机制**，提供一个中心化的存储位置，用于注册、存储和查找对象实例。通常用于需要全局访问、防止重复、统一管理的场景。

**核心特征：**
1. **全局唯一入口**：通过一个静态的注册表类管理所有实例
2. **自动注册**：对象创建时自动注册到注册表
3. **按标识查找**：通过唯一标识（如 ID、名称）查找对象
4. **防止重复**：注册时检查是否已存在，避免冲突

**为什么这是注册表模式而不是单例模式？**
- 单例模式：保证**一个类只有一个实例**，关注的是实例的唯一性
- 注册表模式：管理**多个不同类型的实例**，每个实例有唯一标识，关注的是实例的集中管理和查找

### 实现位置
`cn.dreamtof.core.exception.ErrorCode`

### 示例代码

```java
@Getter
public abstract class ErrorCode implements IErrorCode {
    private static final Map<Integer, ErrorCode> REGISTRY = new ConcurrentHashMap<>();

    protected ErrorCode(int code, String message) {
        this.code = code;
        this.message = message;
        // 自动注册到全局表
        if (REGISTRY.containsKey(code)) {
            throw new IllegalStateException("禁止注册相同错误码: " + code);
        }
        REGISTRY.put(code, this);
    }

    public static ErrorCode getByCode(int code) {
        return REGISTRY.get(code);
    }
}
```

## 3. 工厂模式 + ClassValue 缓存

### 什么是工厂模式？

**定义：** 工厂模式是一种**创建型设计模式**，提供创建对象的接口，由子类决定实例化哪个类。工厂方法让类把实例化推迟到子类。

**核心特征：**
1. **封装创建逻辑**：对象的创建过程被封装在工厂方法中
2. **延迟实例化**：调用方不需要知道具体创建细节
3. **可扩展**：新增产品类型只需新增工厂方法
4. **解耦**：调用方只依赖接口/抽象类，不依赖具体类

**为什么这是工厂模式而不是简单 new？**
- 简单 new：`new FastBeanMeta(User.class)` - 调用方需要知道具体类和构造细节
- 工厂模式：`FastBeanMeta.of(User.class)` - 调用方只知道静态方法，内部可能返回缓存实例或新实例

### 实现位置
`cn.dreamtof.core.utils.FastBeanMeta`

### 示例代码

```java
public class FastBeanMeta {
    private static final ClassValue<FastBeanMeta> CACHE = new ClassValue<>() {
        @Override
        protected FastBeanMeta computeValue(Class<?> type) {
            return new FastBeanMeta(type);  // 工厂方法：延迟创建
        }
    };

    public static FastBeanMeta of(Class<?> clazz) {  // 静态工厂方法
        return CACHE.get(clazz);
    }

    private FastBeanMeta(Class<?> type) {  // 私有构造器
        // 解析类元数据...
    }
}
```

- 比 ConcurrentHashMap 更快，因为它是 JVM 内置的类关联元数据机制

## 4. 模板方法模式（Template Method）

### 什么是模板方法模式？

**定义：** 模板方法模式是一种**行为型设计模式**，在父类中定义算法的骨架（模板），将某些步骤的实现延迟到子类。模板方法让子类在不改变算法结构的情况下，重新定义算法的某些步骤。

**核心特征：**
1. **定义骨架**：父类定义算法的固定流程
2. **钩子方法**：子类可重写的可选方法
3. **抽象方法**：子类必须实现的必要方法
4. **不变与变化分离**：算法结构不变，具体步骤可变

**为什么这是模板方法模式而不是策略模式？**
- 模板方法：**继承关系**，父类控制流程，子类实现细节，流程固定
- 策略模式：**组合关系**，调用方选择策略，策略之间可完全不同，流程可变

### 实现位置
`cn.dreamtof.core.exception.BaseExceptionRegistry`

### 示例代码

```java
public abstract class BaseExceptionRegistry {
    protected final Map<Class<? extends Throwable>, IErrorCode> mappings = new HashMap<>();
    
    // 模板方法：定义固定的算法骨架
    public IErrorCode getErrorCode(Throwable throwable) {
        if (throwable == null) return CommonErrorCode.SYSTEM_ERROR;
        Throwable root = unwrap(throwable);  // 步骤1：钩子方法
        Class<? extends Throwable> clazz = root.getClass();
        IErrorCode cached = cache.get(clazz);
        if (cached != null) return cached;
        IErrorCode match = findMatch(clazz);  // 步骤2：固定步骤
        if (match == null) match = CommonErrorCode.SYSTEM_ERROR;
        cache.put(clazz, match);
        return match;
    }
    
    // 钩子方法：子类可重写
    protected Throwable unwrap(Throwable e) { 
        return e;
    }
}
```

## 5. 策略模式（Strategy Pattern）

### 什么是策略模式？

**定义：** 策略模式是一种**行为型设计模式**，定义一系列算法，将每个算法封装起来，并使它们可以互相替换。策略模式让算法独立于使用它的客户端而变化。

**核心特征：**
1. **算法封装**：每个策略是一个独立的算法实现
2. **可互换**：不同策略可以互相替换
3. **客户端解耦**：客户端不需要知道具体策略的实现细节
4. **运行时选择**：可以在运行时动态选择使用哪个策略

**为什么这是策略模式而不是模板方法模式？**
- 策略模式：**组合关系**，不同策略之间**完全独立**，可以有不同的实现方式
- 模板方法：**继承关系**，子类共享父类的算法骨架，只是某些步骤不同

### 实现位置
`cn.dreamtof.core.utils.SmartValidator`

### 示例代码

```java
private static void performCheck(Object value, Check check) {
    switch (value) {
        case String s -> {
            // String 策略：完全独立的校验逻辑
            if (check.notBlank() && s.isBlank()) {
                throw new BusinessException(check.message());
            }
            if (check.regex().length() > 0 && !s.matches(check.regex())) {
                throw new BusinessException(check.message());
            }
        }
        case Number n -> {
            // Number 策略：完全独立的校验逻辑
            if (check.min() != Long.MIN_VALUE && n.longValue() < check.min()) {
                throw new BusinessException(check.message());
            }
            if (check.max() != Long.MAX_VALUE && n.longValue() > check.max()) {
                throw new BusinessException(check.message());
            }
        }
        case Collection<?> coll -> {
            // Collection 策略：完全独立的校验逻辑
            if (check.notEmpty() && coll.isEmpty()) {
                throw new BusinessException(check.message());
            }
        }
        default -> {}
    }
}
```

## 6. 装饰器模式（Decorator Pattern）

### 什么是装饰器模式？

**定义：** 装饰器模式是一种**结构型设计模式**，动态地给一个对象添加一些额外的职责。就增加功能来说，装饰器模式比生成子类更为灵活。

**核心特征：**
1. **动态增强**：运行时动态添加功能，而不是编译时
2. **不修改原对象**：通过包装的方式增强，不改变原有对象
3. **可叠加**：可以多层装饰，叠加多个功能
4. **透明性**：装饰后的对象与原对象有相同的接口

**为什么这是装饰器模式而不是代理模式？**
- 装饰器：**增加功能**，为对象添加新的行为，关注"增强"
- 代理：**控制访问**，代表另一个对象控制对其的访问，关注"控制"

### 实现位置
`cn.dreamtof.core.context.ContextPropagator`

### 示例代码

```java
public class ContextPropagator implements TaskDecorator {
    @Override
    public Runnable decorate(Runnable runnable) {
        // 捕获上下文
        List<Object> snapshots = new ArrayList<>();
        for (ThreadContextCopier copier : COPIERS) {
            snapshots.add(copier.capture());
        }
        // 返回装饰后的 Runnable（增加了上下文传播功能）
        return () -> {
            for (int i = 0; i < COPIERS.size(); i++) {
                COPIERS.get(i).restore(snapshots.get(i));
            }
            try {
                runnable.run();  // 执行原始任务
            } finally {
                for (ThreadContextCopier copier : COPIERS) {
                    copier.clear();
                }
            }
        };
    }
}
```

## 7. 门面模式（Facade Pattern）

### 什么是门面模式？

**定义：** 门面模式是一种**结构型设计模式**，为子系统中的一组接口提供一个一致的界面。门面模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。

**核心特征：**
1. **简化接口**：提供一个简化的高层接口
2. **隐藏复杂性**：隐藏子系统的复杂实现细节
3. **解耦**：客户端与子系统解耦，只依赖门面
4. **分层**：门面作为子系统的入口层

**为什么这是门面模式而不是适配器模式？**
- 门面：**简化接口**，为复杂子系统提供简单入口，关注"易用性"
- 适配器：**转换接口**，将一个接口转换成另一个接口，关注"兼容性"

## 8. 空对象模式（Null Object Pattern）

### 什么是空对象模式？

**定义：** 空对象模式是一种**行为型设计模式**，提供一个默认的"空"实现，代替 null 值。这个空对象提供"什么也不做"的行为，避免客户端进行 null 检查。

**核心特征：**
1. **避免 null**：用对象代替 null，避免 NullPointerException
2. **默认行为**：空对象提供安全的默认行为
3. **消除判空**：客户端不需要检查 null
4. **可区分**：可以通过标识字段区分空对象和真实对象

**为什么这是空对象模式而不是默认值？**
- 默认值：只是简单的值替换，如 `name = "unknown"`
- 空对象：是一个**完整的对象**，实现了相同的接口，提供安全的行为

### 实现位置
`cn.dreamtof.core.context.OperationContext`

### 示例代码

            .isDefault(true)  // 标识字段：区分空对象和真实对象
            .traceId(UuidCreator.getTimeOrderedEpoch())
            .deviceId("UNKNOWN-DEVICE")
            .startTime(System.currentTimeMillis())
            .businessModule("System")
            .businessAction("Background-Task")
            .clientIp("127.0.0.1")
            .build();
}

## 9. 原型模式（Prototype Pattern）

### 什么是原型模式？

**定义：** 原型模式是一种**创建型设计模式**，通过复制现有实例来创建新实例，而不是通过 new 关键字创建。原型模式通过克隆（Clone）来创建对象。

**核心特征：**
1. **克隆创建**：通过复制现有对象创建新对象
2. **避免构造**：不需要知道对象的具体类和构造细节
3. **深拷贝/浅拷贝**：支持不同级别的复制
4. **性能优化**：对于创建成本高的对象，克隆比 new 更高效

**为什么这是原型模式而不是工厂模式？**
- 工厂模式：**从零创建**，通过构造器创建新对象
- 原型模式：**复制创建**，通过克隆现有对象创建新对象

### 实现位置
`cn.dreamtof.core.utils.FastReflectUtils`

### 示例代码

```java
public static <T> T deepCopy(T source) {
    return deepCopy(source, new IdentityHashMap<>());
}

private static <T> T deepCopy(T source, Map<Object, Object> visited) {
    if (source == null) return null;
    if (isSimpleType(clazz)) return source;
    if (visited.containsKey(source)) return (T) visited.get(source);
    
    // 通过克隆创建新对象，而不是 new
    if (source instanceof Collection<?> coll) {
        Collection<Object> target = (Collection<Object>) newInstance(clazz);
        visited.put(source, target);
        for (Object item : coll) {
            target.add(deepCopy(item, visited));  // 递归克隆
        }
        return (T) target;
    }
    
    T target = (T) newInstance(clazz);  // 克隆目标
    visited.put(source, target);
    for (var accessor : accessors) {
        Object val = accessor.getter().apply(source);
        accessor.setter().accept(target, deepCopy(val, visited));  // 复制属性
    }
    return target;
}
```

## 设计模式总结

| 设计模式 | 核心应用场景 | 关键类 | 判断依据 |
|---------|-------------|--------|---------|
| SPI 模式 | 上下文传播扩展 | ThreadContextCopier | 运行时动态发现实现，接口与实现分离 |
| 注册表模式 | 错误码全局管理 | ErrorCode | 全局注册表管理多个实例，按标识查找 |
| 工厂模式 | Bean元数据缓存创建 | FastBeanMeta | 封装创建逻辑，返回缓存或新实例 |
| 模板方法 | 异常处理流程 | BaseExceptionRegistry | 继承关系，父类定义骨架，子类实现细节 |
| 策略模式 | 类型校验算法选择 | SmartValidator | 组合关系，不同策略完全独立 |
| 装饰器模式 | 异步任务上下文传播 | ContextPropagator | 动态增强功能，不修改原对象 |
| 门面模式 | 反射工具统一入口 | FastReflectUtils | 简化接口，隐藏子系统复杂性 |
| 空对象模式 | 默认操作上下文 | OperationContext | 用对象代替 null，提供安全默认行为 |
| 原型模式 | 对象深拷贝 | FastReflectUtils | 通过克隆创建新对象，而非构造器 |
