# 歌单管理领域（Playlist Domain）需求文档

## 1. 领域概览

| 项目   | 内容                          |
| ---- | --------------------------- |
| 领域名称 | 歌单管理（Playlist）              |
| 聚合根  | Playlist                    |
| 核心职责 | 歌单管理、歌单-歌曲关联                |
| 角色   | 领域事件的消费者                    |
| 处理表  | `playlist`, `playlist_song` |

***

## 2. 领域边界定义

### 2.1 边界内（核心职责）

本领域负责所有与歌单相关的功能：

- **歌单生命周期管理**：创建（MVP自动创建）、更新、软删除
- **歌单状态管理**：公开/私有状态切换
- **歌单-歌曲关联管理**：建立和维护歌曲与歌单的关联关系（消费SongDeletedEvent解除关联）
- **歌单信息展示**：歌单基本信息、歌曲列表展示

### 2.2 边界外（与其他领域的协作）

| 协作领域            | 协作关系     | 说明                                                               |
| --------------- | -------- | ---------------------------------------------------------------- |
| **Auth（身份认证）**  | 订阅领域事件   | 监听 `UserRegisteredEvent`，为用户创建默认歌单                               |
| **Song（音乐内容）**  | **双向协作** | 歌单包含歌曲，通过 `playlist_song` 关联表建立关系；**监听`SongDeletedEvent`解除歌单关联** |
| **Label（标签管理）** | 无直接协作    | 标签由Song领域在添加歌曲时使用，Playlist领域不直接处理标签                              |

### 2.3 领域事件

| 事件名称                  | 类型     | 触发时机                 | 消费者            | 说明             |
| --------------------- | ------ | -------------------- | -------------- | -------------- |
| `UserRegisteredEvent` | 消费     | 用户注册成功（Auth领域发布）     | Playlist领域     | 触发默认歌单创建       |
| `SongDeletedEvent`    | **消费** | **歌曲删除成功（Song领域发布）** | **Playlist领域** | **解除歌曲与歌单的关联** |

***

## 3. 完整边界条件

### 3.1 数据所有权边界

**数据表维护责任**：

- **Playlist领域全权维护**：`playlist`、`playlist_song` 两张表
- **外键关系**（数据库层面约束，领域层校验）：
  - `playlist.user_id` → `user.id`（由Auth领域维护User表）
  - `playlist_song.song_id` → `song.id`（由Song领域维护Song表）
- **歌单所有者定义**：`playlist.user_id` 字段对应的用户即为歌单所有者

**数据所有权原则**：

- Playlist领域对 `playlist` 表拥有完全所有权
- Playlist领域对 `playlist_song` 表拥有完全所有权
- 跨领域的外键引用仅用于数据一致性，所有权不变

### 3.2 领域事件边界

**事件消费清单**：

| 事件                    | 来源领域 | 处理职责          | 幂等性要求           | 失败处理        |
| --------------------- | ---- | ------------- | --------------- | ----------- |
| `UserRegisteredEvent` | Auth | 创建用户默认歌单      | 必须幂等（同一事件可重复处理） | 重试机制 + 告警   |
| `SongDeletedEvent`    | Song | 清理歌曲在所有歌单中的关联 | 必须幂等（可重复执行）     | 异步处理，失败记录日志 |

**事件处理规范**：

- 所有事件处理器必须具备幂等性
- 事件处理失败不应影响源领域业务流程
- 建议使用本地消息表或事件总线保证事件不丢失

### 3.3 服务调用边界

**对外提供服务**（API层暴露）：

- `PlaylistService.getByUserId(userId)`：根据用户ID获取歌单
- `PlaylistService.getBySlug(slug)`：根据slug访问公开歌单
- `PlaylistService.getSongs(playlistId, cursor, limit)`：游标分页获取歌单歌曲

**被其他领域调用**：

- **Song领域**：添加歌曲时需传入 `playlistId`（Song领域负责调用Playlist领域的关联创建逻辑）

**禁止直接调用**：

- ❌ 不直接调用Auth领域的用户服务
- ❌ 不直接调用Label领域的标签服务
- 仅通过事件和ID进行跨领域数据关联

### 3.4 权限校验边界

**权限模型**：基于资源所有者的访问控制

**修改权限**：

- **规则**：仅歌单所有者（`playlist.user_id == currentUserId`）可执行修改操作
- **受保护操作**：更新歌单信息、切换公开/私有状态、从歌单移除歌曲
- **校验时机**：应用层入口处统一校验

**读取权限**：

- **公开歌单（PUBLIC）**：所有人可读（包括未登录用户）
- **私有歌单（PRIVATE）**：
  - 所有者：可读
  - 非所有者（包括登录用户）：返回404（避免信息泄露）

**删除权限**：

- MVP阶段**不支持**删除歌单功能（仅提供软删除标记能力，不开放API）

### 3.5 软删除边界

**软删除策略**：

- `playlist` 表使用 `deleted` 字段（TINYINT）标记软删除状态：0-未删除，1-已删除
- `deleted_at` 字段记录删除时间
- `deleted_by` 字段记录删除操作人（预留）

**关联记录处理策略**（关键决策）：

- **方案1：同步物理删除**（✅ 推荐）
  - 删除歌单时，`playlist_song` 关联记录**物理删除**
  - 理由：关联表无独立业务意义，仅作为歌单与歌曲的关系映射
  - 符合DDD聚合根一致性原则：删除聚合根应同步清理其所有子实体
- **方案2：软删除关联记录**（❌ 不采用）
  - 关联记录也使用软删除，保留历史数据
  - 缺点：增加查询复杂度，关联表无独立生命周期

**最终决策**：采用方案1，在删除Playlist聚合根时，**同步物理删除**所有关联的 `playlist_song` 记录。

### 3.6 乐观锁边界

**乐观锁机制**：

- `playlist` 表增加 `version` 字段（INT类型，默认0）
- 每次更新时 `version + 1`
- 更新条件：`WHERE id = ? AND version = ?`
- 冲突处理：版本不一致时抛出 `OptimisticLockingFailureException`，应用层捕获并提示用户重试

**适用场景**：

- 歌单信息更新（名称、描述）
- 状态切换（公开/私有）
- 高并发场景下的并发控制

### 3.7 关联关系维护边界（核心规则）

`playlist_song` 表是Playlist与Song领域的**共享内核（Shared Kernel）**，必须明确维护边界：

**创建时机**：

- ✅ **Song领域主导**：当用户通过Song模块添加歌曲到歌单时，由Song领域服务调用Playlist领域API创建关联记录
- ❌ Playlist领域**不主动创建**关联（除非未来支持手动添加歌曲功能）

**删除时机**：

| 触发场景   | 删除发起方      | 删除类型 | 说明                           |
| ------ | ---------- | ---- | ---------------------------- |
| 歌曲被删除  | Song领域     | 物理删除 | 监听 `SongDeletedEvent`，清理所有关联 |
| 歌单被删除  | Playlist领域 | 物理删除 | 删除歌单时级联清理关联                  |
| 手动移除歌曲 | Song领域     | 物理删除 | 通过Song模块的移除功能实现              |

**维护责任划分**：

- **创建责任**：Song领域（添加歌曲时）
- **清理责任**：
  - 歌曲删除 → Song领域负责（事件驱动）
  - 歌单删除 → Playlist领域负责（级联删除）
- **查询责任**：两个领域均可查询（通过各自Repository）

**DDD原则依据**：

- 关联表作为共享内核，双方都有读写权
- 但业务行为的发起方应明确：添加歌曲行为属于Song领域，因此Song领域主导创建

### 3.8 ID生成策略

**全局ID策略**：

- 使用 **UUIDv7** 作为所有聚合根的主键
- 字段类型：`CHAR(36)` 或 `BINARY(16)`（推荐后者以节省空间）
- 生成时机：应用层创建聚合根时生成

**优势**：

- 分布式友好，无需中心化ID生成器
- 时间有序，利于索引和分片
- 避免ID猜测和遍历攻击

### 3.9 游标分页边界

**游标设计**：

- **游标组成**：`cursor = Base64(created_at + "_" + id)`
- **排序字段**：`ORDER BY added_at DESC, id DESC`
- **查询条件**：`WHERE (added_at < ?) OR (added_at = ? AND id < ?)`

**分页参数**：

- `limit`：每页大小，建议默认20，最大100
- `cursor`：上一页返回的游标，首页为空

**边界处理**：

- 游标过期：返回空列表，不报错
- 重复数据：使用 `(added_at, id)` 组合确保稳定性
- 新数据插入：不影响历史游标查询结果（基于快照）

### 3.10 与其他领域的数据流转

**数据流向图**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        数据流转边界                                │
└─────────────────────────────────────────────────────────────────┘

Auth领域 ──(UserRegisteredEvent)──> Playlist领域
   │                                      │
   │  用户注册成功                          │  创建默认歌单
   │                                      │
   │                                      ▼
   │                               playlist表插入
   │                                      │
   │                                      │
Song领域 ──(SongDeletedEvent)──> Playlist领域
   │                                      │
   │  歌曲删除成功                          │  清理playlist_song
   │                                      │
   │                                      ▼
   │                               物理删除关联记录
   │                                      │
   │                                      │
Playlist领域 ───(playlistId)─────> Song领域
   │                                      │
   │  提供歌单ID                           │  添加歌曲时使用
   │                                      │
   │                                      ▼
   │                               创建playlist_song
```

**流转规则**：

1. **Auth → Playlist**：单向事件驱动，Playlist领域不依赖Auth领域API
2. **Song → Playlist**：单向事件驱动（SongDeletedEvent）+ 服务调用（添加歌曲）
3. **Playlist → Song**：仅提供ID标识，不调用Song领域业务方法

## 4. 核心领域模型

### 4.1 聚合根：Playlist（歌单）

```
Playlist (聚合根)
├── id: Long                    # 歌单唯一标识（UUIDv7）
├── userId: Long                # 所属用户ID（外键关联User）
├── name: String                # 歌单名称
├── description: String         # 歌单描述
├── status: PlaylistStatus      # 公开/私有状态
├── createdAt: DateTime         # 创建时间
├── updatedAt: DateTime         # 更新时间
├── version: Integer            # 乐观锁版本号
├── deleted: Boolean            # 软删除标记
└── songs: List<PlaylistSong>   # 歌单中的歌曲关联（实体）
```

**业务行为**：

- `createDefault(userId, username)`: 创建默认歌单
- `updateInfo(name, description)`: 更新歌单信息（校验权限）
- `changeStatus(status)`: 切换公开/私有状态
- `isPublic()`: 判断是否为公开状态
- `isOwner(userId)`: 判断是否为歌单所有者
- `markDeleted()`: 标记软删除（MVP不支持，仅预留）
- `removeSong(songId)`: 从歌单移除歌曲（内部方法，由Song领域调用）

**不变性规则**（Invariants）：

- 一个用户只能有一个歌单（`userId` 唯一）
- `userId` 不可变更
- 删除后不可修改

### 4.2 实体：PlaylistSong（歌单-歌曲关联）

```
PlaylistSong (实体)
├── id: Long                    # 关联记录ID（UUIDv7）
├── playlistId: Long            # 歌单ID
├── songId: Long                # 歌曲ID
├── addedBy: Long               # 添加者用户ID
├── addedAt: DateTime           # 添加时间
├── sortOrder: Integer          # 排序顺序（可选）
└── version: Integer            # 乐观锁版本号（可选）
```

**业务行为**：

- `create(playlistId, songId, addedBy)`: 创建关联记录
- `remove()`: 移除关联（物理删除）

### 4.3 值对象

| 值对象                 | 说明   | 校验规则              |
| ------------------- | ---- | ----------------- |
| PlaylistName        | 歌单名称 | 1-50字符，非空，UTF-8编码 |
| PlaylistDescription | 歌单描述 | 可选，最大500字符        |

### 4.4 枚举

```java
public enum PlaylistStatus {
    PUBLIC,     // 公开 - 所有用户可访问
    PRIVATE     // 私有 - 仅所有者登录后可访问
}
```

## 5. 功能需求

### 5.1 自动创建默认歌单（领域事件消费）

- **触发方式**：异步事件驱动
- **事件监听**：监听 `UserRegisteredEvent` 领域事件
- **处理逻辑**：
  1. 接收到用户注册成功事件
  2. 为用户创建默认歌单
  3. 默认名称："xxx的歌单"（xxx为用户名，由前端提供建议）
  4. 默认状态：PUBLIC（公开）
- **幂等性**：通过 `userId` 唯一约束保证，重复事件自动忽略
- **失败处理**：基于事件驱动的可靠性设计，记录失败事件并告警，人工介入补偿

### 5.2 歌单公开/私有设置

- **功能描述**：主播设置歌单的可见性
- **前置条件**：主播已登录，且为歌单所有者
- **权限校验**：`playlist.user_id == currentUserId`
- **主流程**：
  1. 主播在歌单页或设置页切换公开/私有开关
  2. 系统保存设置（乐观锁更新）
- **后置条件**：
  - 公开：所有用户可访问 `/:slug`
  - 私有：其他用户访问返回404，主播自己登录后仍可访问

### 5.3 更新歌单信息

- **功能描述**：主播自定义歌单名称、描述
- **前置条件**：主播已登录，且为歌单所有者
- **权限校验**：`playlist.user_id == currentUserId`
- **主流程**：
  1. 可修改歌单名称（1-50字符）
  2. 可填写歌单描述（最多500字符）
  3. MVP阶段不支持封面图上传
- **并发控制**：使用乐观锁防止并发更新冲突

### 5.4 歌单-歌曲关联管理

- **功能描述**：管理歌曲与歌单的关联关系
- **业务规则**：
  1. 添加歌曲时自动创建关联记录（由**Song领域主导**创建）
  2. MVP阶段每个用户仅有一个歌单，歌曲添加时自动关联
  3. `added_by` 记录添加者用户ID，用于权限追踪
  4. `added_at` 记录添加时间，用于排序
  5. 同一歌曲可在多个歌单中（未来多歌单场景）

### 5.5 歌单列表展示

- **功能描述**：展示用户歌单中的所有歌曲
- **前置条件**：用户已登录（主播）或未登录（普通用户访问公开歌单）
- **权限校验**：
  - 公开歌单：直接返回
  - 私有歌单：校验 `currentUserId == playlist.user_id`
- **主流程**：
  1. 进入歌单页，系统通过游标分页加载歌单歌曲（通过playlist\_song关联查询）
  2. 默认按添加时间倒序排列，最新添加的在最上方
  3. 每条歌曲展示项包含：歌曲名、歌手、标签、备注、付费等级、跳转链接等
  4. 主播登录后，每行显示勾选框，悬停显示编辑/删除按钮
- **异常流程**：
  - 歌单为空：显示空状态提示，引导主播添加第一首歌曲
  - 私有歌单访问：非主播访问私有歌单返回404页面

### 5.6 从歌单移除歌曲

- **功能描述**：从歌单中移除歌曲（保留歌曲记录，仅解除关联）
- **前置条件**：主播已登录，且为歌单所有者
- **实现方式**：通过Song领域提供的移除接口间接实现（删除playlist\_song记录）
- **删除责任**：Song领域负责清理关联

### 5.7 监听歌曲删除事件（领域事件消费）

- **功能描述**：监听 `SongDeletedEvent` 事件，解除被删除歌曲的歌单关联
- **事件来源**：Song领域发布
- **处理逻辑**：
  1. 事件监听器接收 `SongDeletedEvent`
  2. 从事件中获取被删除的歌曲ID
  3. **物理删除** `playlist_song` 表中该歌曲的所有关联记录
  4. 记录操作日志
- **业务规则**：
  - 歌曲删除后，从所有歌单中移除该歌曲
  - 清理操作幂等，可重复执行
  - 异步处理，不阻塞歌曲删除主流程

## 6. 领域事件消费

### 6.1 UserRegisteredEvent 事件处理

```java
// 事件结构
public class UserRegisteredEvent {
    private Long userId;        // 用户ID
    private String username;    // 用户名
    private LocalDateTime registeredAt; // 注册时间
}
```

**处理流程**：

1. 事件监听器接收 `UserRegisteredEvent`
2. 检查是否已存在该用户的歌单（幂等性保证）
3. 调用 PlaylistAppService 创建默认歌单
4. 默认歌单名称："{username}的歌单"
5. 默认状态：PUBLIC
6. 记录操作日志

**幂等性保障**：

- 利用 `playlist.user_id` 唯一约束
- 创建前查询检查，重复事件直接返回成功

### 6.2 SongDeletedEvent 事件处理

```java
// 事件结构
public class SongDeletedEvent {
    private Long songId;        // 歌曲ID
    private Long deletedBy;     // 删除者用户ID
    private LocalDateTime deletedAt; // 删除时间
}
```

**处理流程**：

1. 事件监听器接收 `SongDeletedEvent`
2. 从事件中获取被删除的歌曲ID
3. 删除 `playlist_song` 表中该歌曲的所有关联记录（物理删除）
4. 记录操作日志（删除关联数）

**幂等性保障**：

- 重复删除同一 songId 不影响结果（已删除则影响行数为0）
- 不校验歌单所有权（由Song领域在删除前校验）

**异常处理**：

- 异步处理，捕获异常并记录日志
- 失败事件进入死信队列，人工介入

## 7. 接口清单

| 接口       | 方法  | 路径                       | 说明           | 权限                 |
| -------- | --- | ------------------------ | ------------ | ------------------ |
| 获取我的歌单   | GET | `/playlists/me`          | 获取当前用户的歌单    | 登录用户               |
| 更新歌单信息   | PUT | `/playlists/{id}`        | 更新歌单名称、描述    | 仅所有者               |

## 11. MVP阶段限制与约定

### 11.1 功能限制

| 功能      | MVP支持   | 说明                         |
| ------- | ------- | -------------------------- |
| 歌单创建    | ✅ 自动创建  | 用户注册时自动创建，不支持手动创建          |
| 歌单删除    | ❌ 不支持   | 仅提供软删除标记能力，不开放删除API        |
| 多歌单     | ❌ 不支持   | 每个用户仅有一个歌单（`user_id` 唯一约束） |
| 歌单封面    | ❌ 不支持   | MVP阶段不涉及封面图上传              |
| 手动移除歌曲  | ⚠️ 间接支持 | 通过Song领域删除功能实现             |
| 歌单重命名   | ✅ 支持    | 更新歌单名称                     |
| 歌单描述    | ✅ 支持    | 更新歌单描述                     |
| 公开/私有切换 | ✅ 支持    | 状态切换随意流转                   |
| 游标分页    | ✅ 支持    | 歌单歌曲列表分页                   |

### 11.2 技术限制

- **ID策略**：强制使用 UUIDv7
- **并发控制**：乐观锁
- **删除策略**：物理删除关联记录（级联清理）
- **事件处理**：异步、幂等
- **权限校验**：统一在应用层入口执行

### 11.3 数据一致性保证

- **跨领域一致性**：通过领域事件最终一致性
- **聚合根内一致性**：通过事务保证（单聚合根更新）
- **关联表一致性**：Song领域创建，Playlist领域清理

## 12. 验收标准

1. **注册时自动创建歌单（异步事件驱动机制）**：
   - 触发方式：用户注册成功后发布 `UserRegisteredEvent`，事件监听器异步创建歌单
   - 幂等性：重复事件不重复创建
   - 默认名称："{username}的歌单"
   - 默认状态：公开（PUBLIC）
2. **歌单信息（名称、描述）可更新**：
   - 仅所有者可更新
   - 乐观锁并发控制
3. **MVP阶段每个用户仅有一个歌单，无法创建多个**：
   - `user_id` 唯一约束保证
4. **MVP阶段不支持封面图上传**
5. **歌单公开/私有切换生效**：
   - 公开：所有用户可访问
   - 私有：其他用户访问返回404，主播自己登录后可访问
6. **歌单列表正确展示所有添加的歌曲信息**：
   - 游标分页
   - 按添加时间倒序
7. **领域事件消费正确实现**：
   - 正确监听 `UserRegisteredEvent` 和 `SongDeletedEvent`
   - 事件处理幂等性保证
   - 失败时具备重试或补偿机制
8. **关联表清理正确**：
   - 歌曲删除时，自动清理所有歌单中的关联
   - 删除采用物理删除
9. **权限校验严格**：
   - 修改操作仅限所有者
   - 读取操作根据公开/私有状态控制
